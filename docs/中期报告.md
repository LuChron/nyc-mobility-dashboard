# NYC Urban Mobility Dashboard 项目中期报告

## 1. 项目背景

本项目基于 NYC Taxi & Limousine Commission 公开发布的 Trip Record Data，设计并实现一个交互式纽约城市出行可视化大屏。项目关注 Yellow Taxi、Green Taxi 和 High Volume For-Hire Vehicle（HVFHV，主要对应 Uber / Lyft 等高容量网约车）三类出行服务，尝试从时间变化、空间分布、区域流动和出行行为四个角度分析纽约城市交通出行特征。

纽约 TLC 数据具有较强的时空属性：每条记录通常包含上车时间、下车时间、上车区域、下车区域、行程距离、费用等信息。这些字段既可以支持宏观层面的出行总量分析，也可以支持 Taxi Zone 级别的空间分布和 OD 流向分析。因此，该数据集适合作为课程可视化大作业的数据基础。

考虑到课程项目重点在前端可视化设计与实现，本项目采用“Python 数据预处理 + 静态 JSON 数据 + React 前端大屏”的方式完成整体系统。原始 Parquet 数据先经过脚本清洗和聚合，再导出为前端可直接读取的轻量 JSON 文件；前端不直接读取原始行程明细，而是围绕聚合后的数据完成多视图可视化展示。

当前项目处于中期阶段，已经完成两项主要工作：一是搭建了前端可视化大屏骨架，形成了页面布局、筛选器、地图和主要图表组件；二是完成了原始数据分析、清洗、聚合和前端 JSON 数据导出，为后续接入真实数据和实现图表联动打下基础。

## 2. 分析任务定义

本项目的总体分析目标是：通过交互式可视化方式，展示不同类型纽约出行服务在时间、空间和行为特征上的差异，并帮助用户理解城市内部主要出行热点和流动关系。

围绕该目标，项目设计了四类核心分析任务：

| 分析任务 | 主要问题 | 对应图表或模块 |
|---|---|---|
| 时间差异分析 | 出行需求在不同月份、星期和小时段中如何变化？工作日和周末是否存在不同规律？ | 月度趋势图、星期 × 小时热力图、Peak Hour KPI |
| 空间差异分析 | 哪些 Taxi Zone 是主要出行热点？不同区域的上车和下车需求是否有差异？ | Taxi Zone 地图热力图、Top Pickup/Dropoff 区域排行 |
| 区域流动分析 | 城市内部哪些区域之间存在高频出行连接？机场、曼哈顿核心区等区域的流向是否突出？ | OD 流向线、Top OD Routes 排行 |
| 出行行为分析 | 不同服务类型在费用、距离、时长和出行占比上有什么差异？ | KPI 卡片、服务类型对比图、费用/距离分布图 |

从交互角度看，系统计划支持用户围绕数据源、时间、区域和指标进行筛选。用户可以切换 Yellow Taxi、Green Taxi、HVFHV 或全部服务；可以选择不同年份、月份和工作日/周末；也可以按 borough 或 Taxi Zone 查看局部区域特征。地图区域被选中后，相关指标、排行和 OD 流向将同步变化，从而形成多视图联动分析。

## 3. 数据说明

### 3.1 数据来源与范围

本项目使用 NYC TLC Trip Record Data 中的三类数据：Yellow Taxi Trip Records、Green Taxi Trip Records 和 High Volume For-Hire Vehicle Trip Records。同时使用 Taxi Zone lookup 与 Taxi Zone GeoJSON 作为地理映射数据。数据下载地址：https://www.nyc.gov/site/tlc/about/tlc-trip-record-data.page 。

当前已经完成处理的数据范围如下：

| 项目 | 内容 |
|---|---|
| 时间范围 | 2025-01 至 2026-04 |
| 月份数量 | 16 个月 |
| 数据源 | Yellow Taxi、Green Taxi、HVFHV |
| 原始 Parquet 文件数 | 48 个 |
| 原始数据体积 | 9,089,009,098 bytes，约 9.09 GB |
| 原始记录总量 | 391,817,992 条 |
| 地理数据 | Taxi Zone lookup、taxi_zones.geojson |

按数据源统计，三类数据规模差异明显：

| 数据源 | 文件数 | 原始记录数 | 文件体积 | Schema 稳定性 |
|---|---:|---:|---:|---|
| Yellow Taxi | 16 | 63,600,472 | 1.09 GB | 20 列完全一致 |
| Green Taxi | 16 | 757,466 | 18.5 MB | 21 列完全一致 |
| HVFHV | 16 | 327,460,054 | 7.98 GB | 25 列完全一致 |
| 合计 | 48 | 391,817,992 | 9.09 GB | 分数据源稳定 |

从数据检查结果看，同一数据源内部字段结构稳定，适合按月批量处理；不同数据源之间字段名称不同，但都包含时间、区域、距离和费用等共同语义字段，可以通过统一字段映射支持后续分析。

### 3.2 字段统一

三类数据源的字段名称并不完全一致。例如 Yellow Taxi 和 Green Taxi 的上下车时间字段分别使用 `tpep` 和 `lpep` 前缀，而 HVFHV 使用 `pickup_datetime` 和 `dropoff_datetime`；距离字段在出租车数据中为 `trip_distance`，在 HVFHV 中为 `trip_miles`。因此，数据处理阶段建立了统一字段口径。

| 统一字段 | Yellow Taxi | Green Taxi | HVFHV |
|---|---|---|---|
| `source` | 固定值 `yellow` | 固定值 `green` | 固定值 `hvfhv` |
| `month` | 文件月份 | 文件月份 | 文件月份 |
| `pickup_time` | `tpep_pickup_datetime` | `lpep_pickup_datetime` | `pickup_datetime` |
| `dropoff_time` | `tpep_dropoff_datetime` | `lpep_dropoff_datetime` | `dropoff_datetime` |
| `pickup_zone` | `PULocationID` | `PULocationID` | `PULocationID` |
| `dropoff_zone` | `DOLocationID` | `DOLocationID` | `DOLocationID` |
| `distance` | `trip_distance` | `trip_distance` | `trip_miles` |
| `duration_seconds` | 上下车时间差 | 上下车时间差 | `trip_time` |
| `fare` | `fare_amount` | `fare_amount` | `base_passenger_fare` |
| `tips` | `tip_amount` | `tip_amount` | `tips` |
| `tolls` | `tolls_amount` | `tolls_amount` | `tolls` |

该统一口径使不同服务类型可以进入同一套图表体系中进行比较。

### 3.3 数据质量与清洗口径

原始数据整体质量较好，但仍存在一些需要处理的问题，包括时间越界、非正时长、区域无法映射、非正距离、负费用和极端值等。由于不同图表关注的含义不同，项目没有采用单一删除规则，而是按照图表用途进行分层清洗。

主要清洗口径如下：

| 清洗口径 | 规则 | 用途 |
|---|---|---|
| 时间有效记录 | 上车时间位于文件月份内；下车时间晚于上车时间；时长大于 0 且不超过 24 小时 | KPI、月度趋势、小时需求、星期小时热力图、服务类型对比 |
| 空间区域有效记录 | 上车或下车区域能够映射到 Taxi Zone GeoJSON | 地图热力图、Top 区域排行 |
| OD 有效记录 | 上车和下车区域均能映射到 Taxi Zone，且两者不相同 | OD 流向图、Top OD Routes |
| 费用与距离指标 | 负费用不参与费用统计；非正距离不参与距离统计 | 平均费用、平均距离、费用/距离分布 |

清洗后的主要记录量如下：

| 数据源 | 原始记录数 | 时间清洗后记录数 | OD 清洗后记录数 |
|---|---:|---:|---:|
| Yellow Taxi | 63,600,472 | 62,899,692 | 59,633,162 |
| Green Taxi | 757,466 | 755,134 | 665,094 |
| HVFHV | 327,460,054 | 327,446,414 | 289,226,756 |
| 合计 | 391,817,992 | 391,101,240 | 349,525,012 |

数据处理中记录到的主要异常如下：

| 异常类型 | 记录数 |
|---|---:|
| pickup_time 不在文件月份内 | 561 |
| 非正时长 | 715,717 |
| 时长超过 24 小时 | 475 |
| 上车区域无法映射到 Taxi Zone | 225,948 |
| 下车区域无法映射到 Taxi Zone | 15,113,402 |
| 非正距离 | 1,933,798 |
| 距离超过 100 miles | 32,653 |
| 负费用 | 3,010,791 |
| 费用超过 500 美元 | 9,607 |

这些异常记录并非全部直接删除，而是根据不同分析任务决定是否参与统计。例如，负费用记录不会参与平均费用计算，但仍可保留在出行总量口径之外的清洗说明中；无法映射到 Taxi Zone 的记录不适合绘制地图和 OD 线，但其存在情况会记录在元数据中。

## 4. 设计方案

### 4.1 总体架构

项目采用静态数据驱动的前端可视化架构：

```text
NYC TLC Raw Parquet
        ↓
Python 数据下载、清洗与聚合脚本
        ↓
frontend/public/data/*.json + taxi_zones.geojson
        ↓
React + TypeScript + ECharts 可视化大屏
        ↓
GitHub Pages 静态部署
```

该架构将较重的数据处理工作放在前端运行之前完成。前端只读取聚合后的 JSON 文件，不直接访问原始 Parquet，也不加载行程级明细。这样既能够使用真实大规模数据，又能保证前端页面的运行效率。

### 4.2 前端页面布局

可视化页面采用一屏式多视图布局，整体结构如下：

```text
顶部：项目标题 + 数据更新时间 + 覆盖月份 + 总体指标
左侧：全局筛选器
中部：Taxi Zone 地图热力图 + OD 流向线
右侧：KPI 卡片 + Top 区域排行 + Top OD 路线排行
底部：月度趋势 + 星期小时热力图 + 服务类型对比 + 费用/距离分布
```

该布局将地图作为主视图放在中心位置，用来展示最重要的空间信息；左侧筛选器用于控制全局分析条件；右侧和底部图表提供补充解释，帮助用户从时间、区域、服务类型和出行行为等多个角度理解数据。

### 5.2 数据清洗与聚合实现

数据处理脚本位于 `nyc-mobility-dashboard/scripts/`，当前已经包括：

```text
download_data.py
fetch_taxi_zones.py
preprocess_trips.py
export_frontend_json.py
run_pipeline.py
```

中期阶段实际执行的数据处理命令为：

```bash
python scripts/preprocess_trips.py --threads 6 --memory-limit 10GB --od-top-n 300
```

处理流程如下：

1. 使用 DuckDB 按月、按数据源逐个读取 Parquet 文件。
2. 将 Yellow / Green / HVFHV 映射到统一字段口径。
3. 按时间、区域、OD、费用、距离等图表用途进行清洗。
4. 将每个文件的结果写入 DuckDB 中间聚合表。
5. 全部文件处理完成后导出前端可直接读取的 JSON 文件。

由于原始数据接近 3.92 亿行，直接使用前端或普通内存方式处理都不可行。当前采用 DuckDB 逐文件读取和聚合，能够避免一次性加载全部数据，并且可以只读取可视化所需字段，从而降低内存压力。

## 6. 遇到的困难与解决方案

### 6.1 原始数据规模较大

本项目原始数据共约 3.92 亿条记录，其中 HVFHV 数据量最大，单独达到约 3.27 亿条记录。如果直接在前端读取，页面无法承受；如果在 Python 中一次性读取全部数据，也容易出现内存不足。

解决方案是使用 DuckDB 逐文件处理，每次只读取一个月、一个数据源的数据，并且只读取可视化所需字段。处理过程中直接完成聚合，最终输出轻量 JSON，避免前端接触原始大文件。

### 6.2 三类数据源字段不一致

Yellow Taxi、Green Taxi 和 HVFHV 的字段名称和部分字段含义并不完全一致。例如时间字段、距离字段和费用字段名称不同，HVFHV 的时长字段也与出租车数据不同。

解决方案是在预处理阶段建立统一字段映射，把三类数据整理到共同字段口径下，再进行统一统计。这样后续图表可以用同一套逻辑比较不同服务类型。

### 6.3 异常记录处理需要兼顾准确性和完整性

原始数据中存在时间异常、区域无法映射、负费用、非正距离和极端值等情况。如果全部删除，可能会影响出行总量统计；如果全部保留，又会影响地图、费用和距离分析的准确性。

解决方案是采用分层清洗策略。不同图表根据自身含义使用不同有效记录：时间图表强调时间合法，地图图表强调区域可映射，OD 图表要求上下车区域均可映射，费用和距离图表则过滤对应异常值。

### 6.4 OD 流向图容易过载

Taxi Zone 之间的 OD 组合数量较多，如果全部绘制到地图上，会造成线条拥挤，也会影响页面性能和阅读效果。

解决方案是在数据导出阶段按 `source + month + day_type` 保留 Top OD 路线。前端展示时也计划只绘制 Top N 线路，使地图呈现重点流向，而不是堆叠所有连接。

### 6.5 多视图联动逻辑仍需完善

当前前端已经完成界面骨架，但真实数据尚未全面接入。后续需要让筛选器、地图、KPI、趋势图、热力图和排行表共享同一组筛选状态。

解决方案是下一阶段建立统一数据加载和筛选逻辑，将清洗后的 JSON 数据转换为当前图表所需的数据结构，并逐步替换现有 mock 数据。
